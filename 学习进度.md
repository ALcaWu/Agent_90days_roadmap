### Day 09 心得
今天学习了ChatModel与多模型集成，重点掌握了：
1. **ChatModel vs LLM 的本质区别**：LLM 接收纯文本字符串，而 ChatModel 接收有角色语义的消息列表（HumanMessage / SystemMessage / AIMessage），ChatModel 更贴近现代对话 AI 的使用方式，也是实际开发中的首选
2. **LangChain 统一接口的价值**：无论是 OpenAI、Anthropic（Claude）还是 HuggingFace，调用方式完全一致，只需替换 `ChatXxx` 类即可切换模型——这让业务代码与具体模型解耦，降低了日后迁移和对比测试的成本
3. **模型工厂模式的工程意义**：通过 `create_chat_model(provider, model)` 工厂函数统一创建入口，可以做到"配置驱动"，根据不同环境（开发/生产）或成本要求动态切换模型，避免硬编码
4. **模型选择的成本意识**：gpt-3.5-turbo 与 gpt-4 之间存在约 15 倍的成本差距，在开发调试阶段使用便宜模型、在生产关键路径上使用强模型，是 Agent 开发中非常重要的工程决策

### Day 10 心得
今天学习了 Prompt Template 与 LCEL，重点掌握了：
1. **PromptTemplate 的核心价值**：将硬编码的字符串拼接升级为"模板+变量"模式，提升了复用性和可维护性，`partial()` 方法可以预填充部分变量，非常适合做角色固定+任务可变的场景
2. **ChatPromptTemplate 的多角色设计**：通过 SystemMessage 定义 AI 的行为边界，HumanMessage 传递用户输入，`MessagesPlaceholder` 动态插入对话历史——这是实现多轮对话的基石
3. **LCEL 管道语法的优雅之处**：`template | model | parser` 一行代码就构成了完整的 AI 调用链，每个组件都是可独立替换的积木，这种声明式的写法比命令式的函数调用更清晰、更易扩展
4. **`partial()` 与依赖注入**：通过 `partial_variables` 在构建模板时注入格式指令（如 `parser.get_format_instructions()`），让模板与解析器形成契约，是 Day 11 Output Parser 的重要铺垫

### Day 11 心得
今天学习了 Few-shot 提示工程与 Output Parser，重点掌握了：
1. **Few-shot 的本质是"示范教学"**：与其用复杂的语言描述你想要什么格式，不如直接给几个例子——Few-shot 让模型通过模仿示例来输出，比 Zero-shot 在格式控制上可靠得多，尤其适合输出结构固定的场景
2. **PydanticOutputParser 是生产级首选**：相比 `JsonOutputParser` 只返回 `dict`，`PydanticOutputParser` 返回有完整类型约束的 Pydantic 对象，IDE 有自动补全，运行时有类型验证，`Field(ge=0)` 等约束还能防止 LLM 输出非法值，这才是工程化的正确姿势
3. **RunnableParallel 的并行威力**：用一行代码同时执行多个分析任务并合并结果，比串行调用高效得多，非常适合"同时需要多个角度分析同一段文本"的场景（如同时提取摘要、关键词、情感）

### Day 12 心得
今天完成了 Output Parser 全部5道练习，重点掌握了：
1. **自定义 Parser 的核心模式**：继承 `BaseOutputParser`，实现 `parse()` 和 `get_format_instructions()` 两个方法即可。正则表达式是做结构化解析的核心工具——练习3用正则提取 Markdown 任务列表的完成状态、优先级、日期，最后通过字符串切片 `content[:match.start()].strip()` 把这些标记从标题里精准删掉，细节决定体验
2. **框架版本陷阱**：课程笔记里的 `RetryOutputParser` 和 `XMLOutputParser` 都有问题——前者已从 `langchain_core` 移除，后者需要额外依赖 `defusedxml`。在真实项目中动手验证非常重要，不能只看文档照搬
3. **PyCharm IDE 层错误排查**：pytest 在收集阶段报错指向一个项目外的损坏文件（WinError 1337 安全描述符无效），堆栈误导性很强，以为是代码问题，实际上是 IDE 的索引/扫描机制碰到了权限损坏的路径，与项目代码完全无关
4. **LCEL 组件的可组合性**：`RunnableLambda` 可以把任何普通函数变成链的一部分，这让我意识到 LCEL 不只是调用 LLM，而是一个通用的"数据处理管道"——任何可以用函数表达的转换逻辑，都可以无缝融入链中

**练习题踩坑总结：**
- **FewShotPromptTemplate 参数拼写**：`perfix` 是不存在的参数，正确拼写是 `prefix`——这个错误会导致整个模板构造失败，调试时很容易被忽略
- **异常处理必须包裹整个调用链**：`parser.invoke()` 写在 try 外面是无效的，一旦 LLM 返回非法内容，异常仍会在 invoke 时抛出，必须把整个调用包进 try-except
- **OutputParser 不做额外格式化**：如 `CommaSeparatedListOutputParser` 不会自动去除列表元素的首尾空格，需要手动 `[s.strip() for s in ...]`
- **中文检测不能靠字面量**：`"中文" in x` 只能检测包含"中文"这个特定字眼的字符串，检测普通中文字符应该用正则 `re.search(r'[\u4e00-\u9fff]', x)`

### Day 13 心得
今天完成了 Day 13 练习题1-3题的全部修复与验证，重点掌握了：
1. **PydanticOutputParser 的正确用法**：返回的是 Pydantic 对象，需 `model_dump()` 才支持字典式访问；`year` 字段用 `Field(ge, le)` 而非 `Annotated`；`@field_validator` 支持 `mode="before"` 做容错预处理
2. **BaseOutputParser 是抽象类**：`RobustJsonParser` 继承它后必须实现 `parse()` 方法，否则无法实例化
3. **单引号键的正则优先级**：JSON 中 `'name': '张三'` 要先修键的单引号再修值的单引号，顺序不能反；否则键的单引号残留导致 `json.loads` 失败
4. **正则匹配顺序与优先级判断**：多条件 `if/elif` 中更具体的条件要放前面（如 `"无优先级"` 要单独判断，不能默认映射为 `"中"`）；截止日期用 `date.fromisoformat().isoformat()` 转字符串
